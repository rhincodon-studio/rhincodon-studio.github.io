Observability

1. Introduction​

可觀測性（Observability, O11y）是現代 DevOps 與 SRE 的核心能力。 在高度複雜的系統中，服務通常是分散、多語言、事件驅動、跨 Kubernetes Node 運行，傳統 Monitoring 已無法足以回答：

  • 為什麼服務變慢？
  • 為什麼某些 Request 只在特定路徑上出問題？
  • 哪個節點、哪條 Trace、哪段程式碼正在造成瓶頸？

Observability 的核心目的，是讓工程團隊能夠在複雜系統中「自由提問」，並從系統的輸出（Logs / Metrics / Traces）推論出內部狀態。 與傳統 Monitoring 僅能回答「是否壞掉」不同，Observability 能夠協助回答「為什麼壞掉」與「還會在哪裡壞」。

2. What is Observability?（什麼是可觀測性）​

Observability 最早源自控制理論，指：

僅透過輸出（Output）能否推斷系統內部狀態（Internal State）

在現代軟體工程領域，Observability 被重新定義為：

  • 能夠探索未知問題
  • 系統在沒有預先定義 dashboard 的前提下仍能回答工程師想問的問題
  • Logs、Metrics、Traces 並非目標，而是觀測能力的「原料」

現代 Observability 常被定義為三種能力的集合：

  • Explorability（可探索性）：面對未知問題時能提出新問題並立即得到答案。
  • Diagnosability（可診斷性）：能快速定位問題根因，而不需猜測或大量試誤。
  • Predictability（可預測性）：能從系統行為推估未來可能的異常、風險與瓶頸。

Observability 因此不是工具堆疊，而是一種探究系統內部狀態的能力模型。

3. Observability vs Monitoring（觀測性 vs 監控）​

Monitoring（監控）Observability（可觀測性）
針對已知問題設計（threshold、alert）面對未知問題也能探索根因
圖表固定、問題固定可任意提問、任意切片查詢
偵測是否壞掉了解「為什麼」壞掉
著重事件發生後告警著重問題探索與分析

Observability 包含 monitoring，但不等於 monitoring。

4. O11y 三大支柱（Three Pillars）​

4.1 Logs（日誌）​

用來記錄事件、狀態變化、錯誤訊息。

4.2 Metrics（指標）​

以數值方式量化系統行為，可用於告警與趨勢分析。

4.3 Traces（追蹤）​

跨服務、跨 Pod、跨 Node 將一次請求的完整生命週期串起。

近年更流行的觀念 → 三支柱不是目標；目標是「能回答問題」。

4.4 OpenTelemetry Telemetry Model（統一遙測模型）​

2023 年後，OpenTelemetry 將 Logs / Metrics / Traces 視為同一個 Telemetry Model 的三種事件表現形式，並透過「Resource、Attributes、Context」統一定義所有遙測資料。 這代表三大支柱不再是分離的資料源，而是可共同查詢、共同建模的事件流。

這讓跨 Logs → Metrics → Traces 的關聯查詢成為可能，也是現代 Observability 的核心特色。

6. Observability Stack（實務技術架構）​

以下為現代 O11y 常見架構：

6.1 Kubernetes Observability​

  • Prometheus：Metrics
  • Loki / Elasticsearch：Logs
  • Jaeger / Tempo：Traces
  • OpenTelemetry Collector：資料管道核心
  • Grafana：視覺化

6.2 應用程式 Observability​

  • Java / Spring Boot：OTel Java agent、Micrometer
  • Node.js：OTel SDK
  • Python：OTel instrumentation libraries

6.3 雲端平台支援​

  • GCP、AWS、Azure 原生整合 OTel

7. Modern Observability Principles（現代可觀測性原則）​

✔ 能回答任何問題（The ability to ask any question） Dashboard 不是主角，資料的可組合性才是重點。

✔ 探索未知問題（Unknown-unknowns） 不是只有 alerting，而是分析 → 假設 → 驗證 → root cause。

✔ 高維度、高基數資料（High-Cardinality / High-Dimensionality） 例如：

  • user_id
  • trace_id
  • environment
  • pod_id
  • feature_flag

8. Summary（總結）​

Observability 不是工具，而是一種方法論：

  • 控制理論起源（1960 Kálmán）
  • Honeycomb 與 Charity Majors 推入工程實務（2018）
  • OpenTelemetry 彙整與標準化（2019–Now）

在雲原生架構下，Observability 是 DevOps / SRE 的核心能力， 能讓你理解系統真正發生了什麼事、為什麼會發生，以及如何避免再次發生。
